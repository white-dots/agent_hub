How the benchmark works

1. Generate synthetic codebases

Three Python projects of increasing size (10, 50, 200 modules) with realistic dependency structures:

  • Source modules organized into packages (core/, services/, api/, utils/, models/)
  • Hub modules — files imported by 20–40% of other modules (the high-risk files)
  • Random cross-module dependencies filling out the graph
  • Test files that import source modules (~20% of total)

2. Simulate breaking edits

For each hub module, we simulate mutations that change the public interface:

  • Rename function — process_data() → process_data_v2()
  • Change signature — add/remove a parameter
  • Rename class — UserService → UserServiceV2

4. Compare scenarios

  • Without tool: Developer edits blindly. Broken dependents discovered later via test failures or runtime errors.
  • With impact_check: Tool walks import graph (BFS over imported_by edges), surfaces every affected file. Developer updates them before committing.

"We simulated 64 code edits across codebases of 10–200 modules. Without impact analysis, 56% of edits to hub files silently broke downstream code — an average of 36 files per breaking edit. With impact_check, 100% of that breakage is surfaced before it reaches production. Sub-millisecond. Zero LLM cost."
