Why AgentHub Architecture Benchmarks Token Efficiency

Why AgentHub Exists

Vanilla Claude Code reads your entire codebase into its context window on every question. AgentHub gives it specialized agents that already know their domain — so it doesn't have to.

The Problem

What Happens as Codebases Grow

Claude Code is extraordinary at navigating code. But it has a fundamental constraint: every question starts with a blank slate. It must re-discover your codebase structure, re-read relevant files, and re-build understanding from scratch — every single time.

⚠ Vanilla Claude Code

  • 🔍 Searches the entire codebase for every question
  • 📈 Burns 200K–1.2M tokens per complex question
  • ⏳ Spends 30–180s just finding the right files
  • 🗃 No persistent memory of code structure between sessions
  • 🚧 Hits context window limits on large codebases (100K+ LOC)
  • 🎯 Can miss relevant files buried in deep directory trees
vs

✓ Claude Code + AgentHub

  • 🎯 Routes queries directly to the agent that owns that domain
  • 📈 Agents carry pre-built context (~5–50K tokens each)
  • ⚡ Instant domain knowledge — no file scanning needed
  • 🧠 Persistent understanding survives across sessions
  • 📚 Scales to any codebase size via agent decomposition
  • 👥 Cross-cutting queries handled by multi-agent DAG teams

The core insight

A human engineering team doesn't read the entire codebase to answer a question about the pricing module. They ask the person who owns that module. AgentHub gives Claude Code the same organizational structure — specialized agents that already know their domain, so Claude can ask instead of search.

Where AgentHub Adds the Most Value

Large Codebases

50K+ LOC codebases where vanilla Claude Code spends most of its tokens and time just navigating. Agents eliminate the search phase entirely.

Cross-Cutting Questions

"How does data flow from API to database to frontend?" Questions that span multiple modules get decomposed into sub-questions for each domain agent, then synthesized.

Parallel Implementation

Multi-part requests like "Add a new API endpoint AND a new UI component" get split into independent tasks running simultaneously on separate git branches.

Architecture

A Company Model for Code Understanding

AgentHub models a software company. The CEO (you) makes high-level decisions. The CTO (Claude Code) orchestrates. Business leads (Tier A) know the "why." Team leads (Tier B) know the "how." Each agent carries context so Claude doesn't have to rebuild it.

CEO — You (the developer)
↓ High-level decisions only
CTO — Claude Code + AgentHub MCP
↓ Routes queries to the right team
Tier A — Business Agents
Manually created · Know the "why"
Tier B — Code Agents
Auto-generated · Know the "how"
Tier C — QC Agents
Automated · Review changes
Tier A: You write these to encode business logic (pricing strategies, API integrations, domain rules).
Tier B: AgentHub auto-generates these by analyzing your codebase's import graph and file structure.
Tier C: Automatically review code changes for breaking changes, security issues, and missing tests.

Zero-Config Agent Generation

Run agenthub build . and Tier B agents are created automatically. AgentHub scans your codebase, builds import graphs (Python AST + TypeScript regex), detects natural module clusters, and creates one agent per domain. Agents that grow too large (60+ files) are automatically split into sub-agents.

Intelligent Routing

Every query is routed to the best agent using keyword matching with IDF weights, tier-aware classification (code terms → Tier B, business terms → Tier A), and complexity detection. Simple queries go to one agent (2–5s). Cross-cutting queries ("trace the data flow from API to DB") trigger DAG team execution with multiple agents in parallel, then synthesis.

Root Cause Analysis

Why AgentHub scores 0.06–0.09 on 2 questions

Routing failures. "Permission engine" and "Promotion rules" are rejected by all agents via heuristic keyword check. After 5 retry attempts, AgentHub gives up with a "not my domain" response. The core issue: no agent's keywords match the specific terminology (e.g., "CASL", "promotion engine"). Fix: broader keyword coverage + LLM-based fallback routing.

Where AgentHub ties or wins

When routing works, quality is comparable. On 5/11 questions where agents weren't scope-rejected, AgentHub scored 0.80–0.93 (vs Claude's 0.81–0.93). AgentHub was 2.2x faster on Medusa Q1 (81s vs 181s) because the relevant agents already had cart/order context loaded. The inventory reservations question (0.92 vs 0.87) shows AgentHub can exceed Claude when routing is correct.

Honest Analysis

AgentHub advantage: speed

40% faster wall-clock time on both repos despite using more tokens. Strapi: 455s vs 634s (0.72x). Medusa: 264s vs 512s (0.52x). This is because AgentHub agents carry pre-built context and run team members in parallel, while Claude Code searches the entire codebase sequentially.

AgentHub disadvantage: token cost

77x more API tokens per question (Strapi: 87K avg vs Claude's ~1.1K output-only). The overhead comes from running 3–5 agents per question in team mode, each with its own context window. When routing fails (2/11 questions), tokens are wasted on scope checks across all agents before giving up.

Bottom line

AgentHub trades API tokens for wall-clock speed and specialized depth. It's faster (0.6x) and matches Claude's accuracy when routing works (5 of 11 questions scored within 0.05), but routing failures tank 2 questions badly. The path to improvement is clear: fix routing to eliminate the 0.06–0.09 scores, which would bring the overall average from 0.56 to ~0.75+. The token-for-speed tradeoff is favorable for teams that value developer time over API cost.

Evaluation Method

  • Condition A: claude -p <question> in repo directory
  • Condition B: hub.run(question) via Python API
  • Scoring: Automated — file_coverage (40%) + keyword_match (40%) + length (20%)
  • Ground truth: Hand-written expected answers with relevant file paths
  • Tokens (B): Full input+output from Anthropic API
  • Tokens (A): Output-length estimate only (not full API tokens)

Limitations

11 matched A/B questions across 2 repos. Results are directional, not statistically significant. Claude Code token counts are output-only estimates (actual usage is 50–200x higher). Two routing failures (scoring 0.06–0.09) heavily skew AgentHub's average downward. Automated scoring (keyword+file match) doesn't capture answer quality nuances — both conditions often produce equally thorough answers that score differently due to exact file path mentions.
